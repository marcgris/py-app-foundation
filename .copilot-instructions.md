# Copilot Instructions for Py App Foundation

GitHub Copilot agents working in this repository should follow these guardrails and workflows.

## Core Guardrails

1. **Always Plan First**: Before implementing changes, write a brief comment or docstring explaining your approach
2. **Write Tests First**: Never implement features without corresponding tests
3. **Validation Must Pass**: All changes must pass `ruff check`, `ruff format`, `pyright`, `pytest`, and `bandit`
4. **Never Merge**: You may create PRs, but humans must review and merge
5. **Respect Boundaries**: Do not modify CI workflows, agent instructions, or planning documents without explicit approval

## Workflow

1. Read the issue or feature request carefully
2. Check [docs/plan/ROADMAP.md](docs/plan/ROADMAP.md) to understand current priorities
3. Review [docs/plan/DECISIONS.md](docs/plan/DECISIONS.md) for architectural context
4. Study the relevant existing code in `src/starter/` and `tests/`
5. **Comment your plan** before implementing
6. Write tests in `tests/unit/` or `tests/integration/` that specify the desired behavior
7. Implement the feature to pass the tests
8. Run all validation locally—do not open a PR until everything passes
9. Create a PR with:
   - Clear title describing the change
   - Description explaining the goal and design
   - Comment on your own PR explaining decisions and trade-offs
10. Await human review and approval

## Allowed Operations

✅ Create or modify files under `src/starter/` and `tests/`  
✅ Update configuration in `pyproject.toml` (with justification)  
✅ Write documentation in `docs/guide/`  
✅ Create new test fixtures or patterns in `tests/conftest.py`  

## Forbidden Operations

❌ Modify `.copilot-instructions.md` or `.instructions.md` without approval  
❌ Modify CI workflow files under `.github/workflows/`  
❌ Modify planning documents under `docs/plan/`  
❌ Add dependencies without documenting why in the commit message  
❌ Merge PRs or push directly to main  
❌ Commit work that fails any validation check  

## Code Standards

- **Type Hints**: Required everywhere (pyright in strict mode)
- **Docstrings**: Google style for all public functions and classes
- **Tests**: Minimum 80% coverage; unit tests for core logic, integration tests for cross-module interaction
- **Naming**: Clear, searchable names; avoid generic terms like `util`, `helper`, `misc`
- **Imports**: Organized, deduplicated; let ruff sort automatically
- **Line Length**: 100 characters max (configured in pyproject.toml)

## Validation Commands

Before opening a PR, run locally:

```bash
uv sync                                          # Install dependencies
uv run ruff check . && uv run ruff format .      # Lint and format
uv run pyright src/                              # Type-check
uv run pytest tests/ -v --cov=src/starter        # Test with coverage
uv run bandit -r src/                            # Security scan
```

All must pass with no exceptions.

## When Stuck

1. Check the issue comments for context
2. Read the relevant docstrings and type hints in the code
3. Look at existing tests to understand patterns
4. Review [docs/guide/contributing.md](docs/guide/contributing.md) for patterns
5. If unresolved, comment on the issue with your question and stop
6. Do not guess about design intent—ask first

## Examples of Good Agent Work

- ✅ Add a test, implement the feature, run all validation, open a PR with clear explanation
- ✅ Refactor an existing module, ensure all tests still pass, explain the improvement in PR comment
- ✅ Add a new pattern to the starter, create tests demonstrating usage, document in guide

## Examples of Bad Agent Work

- ❌ Implement features without tests
- ❌ Open a PR without running validation
- ❌ Merge your own code
- ❌ Add dependencies without justification
- ❌ Modify agent instructions or CI without approval
- ❌ Commit code that fails checks and "fix it later"

Remember: Small, focused PRs with clear intent are easier to review and better for team workflow. Plan, test, validate, explain—then wait for approval.
