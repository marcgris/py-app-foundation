# Py App Foundation Repository Instructions

This document defines the repository conventions, coding standards, and agent behavior for the Py App Foundation project.

## Code Organization

```
src/starter/              # Core reusable patterns
  __init__.py            # Package identity
  config.py              # Typed environment settings
  logging.py             # Logging bootstrap
  exceptions.py          # Custom exceptions

tests/
  conftest.py            # Shared fixtures
  unit/                  # Fast isolated tests
  integration/           # Multi-component tests
```

## Naming Conventions

- **Modules**: Use clear, single-purpose names (config, logging, exceptions, not utils or helpers)
- **Functions/Classes**: Use verbs for functions (load_config, set_up_logging), nouns for classes
- **Test Files**: test_<module>.py pattern (test_config.py, test_logging.py)
- **Fixtures**: Use descriptive names (mock_config, temp_config, default_logger)
- **Variables**: Avoid single letters except in obvious contexts (i in loops, e for exceptions)

## Code Patterns

### Configuration Pattern (config.py)

```python
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    """Application settings, loaded from environment."""
    
    debug: bool = False
    log_level: str = "INFO"
    
    class Config:
        env_file = ".env"
        case_sensitive = False
```

Use in code:

```python
from starter.config import Settings

settings = Settings()
```

### Logging Pattern (logging.py)

```python
import logging

def get_logger(name: str) -> logging.Logger:
    """Get a logger for the given module name."""
    return logging.getLogger(name)

def setup_logging(level: str = "INFO") -> None:
    """Configure application logging."""
    logging.basicConfig(
        level=getattr(logging, level),
        format="%(asctime)s - %(name)s - %(levelname)s - %(message)s"
    )
```

Use in code:

```python
from starter.logging import get_logger

logger = get_logger(__name__)
logger.info("Application started")
```

### Exception Pattern (exceptions.py)

```python
class StarterError(Exception):
    """Base exception for starter errors."""
    pass

class ConfigError(StarterError):
    """Raised when configuration is invalid."""
    pass
```

Use in code:

```python
from starter.exceptions import ConfigError

try:
    # Load config
except ValueError as e:
    raise ConfigError(f"Invalid config") from e
```

### Test Pattern (tests/conftest.py)

```python
from pytest import fixture

@fixture
def mock_config():
    """Provide a test configuration."""
    from starter.config import Settings
    return Settings(debug=True, log_level="DEBUG")
```

Use in tests:

```python
def test_logging_with_config(mock_config):
    """Test that logging respects config."""
    from starter.logging import setup_logging
    setup_logging(mock_config.log_level)
    # assertions...
```

## Type Hints

Every function must have type hints:

```python
from typing import Any, Optional

def load_config(path: Optional[str] = None) -> dict[str, Any]:
    """Load configuration from file or environment."""
    pass

async def process(data: bytes) -> str:
    """Process data asynchronously."""
    pass
```

Use `|` syntax for unions in Python 3.14+:

```python
def get_value(key: str) -> int | None:
    """Get optional value."""
    pass
```

## Docstrings

Use Google style for all public APIs:

```python
def calculate_sum(values: list[int]) -> int:
    """Calculate the sum of values.

    Args:
        values: A list of integers to sum.

    Returns:
        The sum of all values.

    Raises:
        ValueError: If values is empty.
    """
    if not values:
        raise ValueError("values cannot be empty")
    return sum(values)
```

## Testing Standards

- **Minimum Coverage**: 80% across src/starter/
- **Test Types**: Unit tests for individual functions/classes, integration tests for module interaction
- **Test Organization**: One test file per module (test_config.py, test_logging.py)
- **Assertions**: Use clear, specific assertions (not just "assert True")
- **Fixtures**: Shared fixtures go in conftest.py; module-specific fixtures in test file

Example test:

```python
def test_settings_loads_from_env(monkeypatch):
    """Test that Settings loads values from environment."""
    monkeypatch.setenv("DEBUG", "true")
    settings = Settings()
    assert settings.debug is True

def test_invalid_settings_raises_error():
    """Test that invalid settings raise validation error."""
    from pydantic import ValidationError
    with pytest.raises(ValidationError):
        Settings(log_level="INVALID")
```

## Import Organization

Organize imports in this order:

1. Standard library imports
2. Third-party imports (pydantic, pytest, etc.)
3. Local imports (from starter or tests)

Use ruff to manage sorting:

```python
import logging
from typing import Any, Optional

from pydantic import BaseModel
from pydantic_settings import BaseSettings

from starter.config import Settings
from starter.logging import get_logger
```

## Error Handling

Always be explicit about error handling:

```python
def load_config(path: str) -> dict[str, Any]:
    """Load configuration, raising ConfigError if invalid."""
    try:
        with open(path) as f:
            return json.load(f)
    except FileNotFoundError as e:
        raise ConfigError(f"Config file not found: {path}") from e
    except json.JSONDecodeError as e:
        raise ConfigError(f"Invalid JSON in config: {path}") from e
```

Never silently catch exceptions without logging or re-raising:

```python
# ❌ BAD: Silent exception
try:
    process()
except Exception:
    pass

# ✅ GOOD: Explicit handling
try:
    process()
except ConfigError as e:
    logger.error(f"Configuration error: {e}")
    raise
except Exception as e:
    logger.error(f"Unexpected error: {e}")
    raise
```

## Validation and Ci

Before committing, always run:

```bash
uv run ruff check . && uv run ruff format .  # Lint and format
uv run pyright src/                          # Type-check
uv run pytest tests/ -v                      # Tests
uv run bandit -r src/                        # Security
```

No exceptions. CI will enforce all checks; PRs cannot merge if any check fails.

## Scope Constraints for Agents

### applyTo Patterns

- `src/starter/**` and `tests/**`: Agents can modify freely within these directories
- `.github/workflows/**`: Read-only; agent cannot modify CI
- `docs/plan/**`: Read-only; planning documents owned by humans
- `docs/guide/**`: Agents can update if explaining new patterns

### Change Size Limits

- Max 5 files per PR
- Max 300 lines changed per PR (excluding tests and docs)
- Each PR should be one logical feature or fix
